# Touch Portal Hardware Monitor 2.2.1

A patch release focused on **FPS accuracy** and **robustness fixes** on top of [2.2.0](https://github.com/spdermn02/TouchPortal-HardwareMonitor/releases/tag/v2.2.0). No breaking changes — settings and state IDs from 2.2.x are preserved.

## ✨ New

- 🎯 **PresentMon FPS backend** — a new, accurate FPS source powered by Intel [PresentMon](https://github.com/GameTechDev/PresentMon) (bundled). Far more reliable than the in-process ETW counter.
- The **`FPS Source`** setting is now `Off / RTSS / PresentMon / Built-in / Auto`. **`Auto`** prefers **RTSS** (if running) → **PresentMon** → **Built-in ETW**.

## 🐛 Fixes

- 🎮 **FPS "Built-in" reported wildly wrong values** (e.g. bouncing 15 → 1500). The ETW counter now counts one event per present instead of double-counting present start/stop and other events. It's still an *approximate* present-call rate — prefer RTSS or PresentMon for accuracy.
- 🖥️ **`dwm.exe` (desktop compositor) is no longer reported as FPS.** On the desktop with no game running, the FPS state now reads empty instead of a phantom ~refresh-rate number.
- 🌍 **Non-English hardware now shows up.** Sensor names with accents or symbols (e.g. a French water-cooler's `T° Eau 360`, `Débitmetre`) produced invalid state IDs that Touch Portal rejected, so those states never appeared. State IDs are now ASCII-safe (accents transliterated, symbols dropped); the original name still shows in the state description. **Existing state IDs are unchanged.**
- 🧹 **Control characters stripped from hardware/sensor names** — garbled strings (e.g. Crucial DIMM part numbers with embedded NUL/form-feed) no longer leak into the log or state IDs.
- 🔁 **Re-importing the plugin while FPS is active no longer fails.** PresentMon now runs from a working copy under `%AppData%`, and is force-terminated with the plugin (kill-on-close job object), so the plugin folder is never file-locked during import.

## 🛠️ Diagnostics

- **DEBUG logging for FPS** — with `loglevel.txt = DEBUG`, the log shows the chosen backend, why a backend did/didn't start, and each FPS read (value/source/app).
- CLI probes: `TouchPortalHardwareMonitor.exe --fps` (sample FPS ~10s) and `--displays` (list monitors).

## 📌 Notes

- **FPS backends & Admin:** RTSS needs no admin; **PresentMon and Built-in ETW require the plugin to run elevated** (it already does via its launcher). `Auto` falls back gracefully.
- **Built-in ETW FPS is approximate** and still experimental — it over-reads uncapped apps. For trustworthy numbers, run RTSS/MSI Afterburner or use the PresentMon source.
- **Serial coolers (e.g. Aquacomputer Heatmaster):** these are detected by LibreHardwareMonitor at startup over a COM port. If another app holds the port, or the device isn't connected when the plugin starts, it won't appear until the port is free and the plugin is restarted.

## 📥 Install

Import `TouchPortalHardwareMonitor-Windows-2.2.1.tpp` via **Touch Portal → Settings → Plug-ins → Import plug-in…**, and accept the UAC prompt. See the [README](https://github.com/spdermn02/TouchPortal-HardwareMonitor#readme) for full setup and troubleshooting.
